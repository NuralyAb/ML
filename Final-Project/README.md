# Med Forecast KZ — прогноз выписки рецептов по регионам РК

ML-система прогнозирования количества выписанных рецептов на следующие
месяцы в разрезе **регион × район × диагноз (МКБ-10)** на основе
исторических данных регистра рецептов **ЭРСБ МЗ РК** (датасет
`ISLO_MEDICALHISTORYOFCITIZENS` на портале
[ashyq.data.gov.kz](https://ashyq.data.gov.kz)).

> Пользователь — Министерство здравоохранения РК. Прогноз помогает
> принимать решения по закупке препаратов, распределению кадров и
> аудиту поликлиник.

---

## 1. Постановка задачи

**Цель.** Прогноз `recipe_count` — числа рецептов в конкретном месяце
по конкретному коду МКБ-10 в конкретном регионе/районе.

**Гранулярность.** На уровне модели — `(region × icd_code × month)`
(уровень района слишком разрежён для классической регрессии: многие
комбинации имеют < 12 наблюдений). Уровень района сохраняется отдельно
для дашборда (фильтры, агрегаты).

**Горизонт.** 1, 3, 6 или 12 месяцев вперёд (рекурсивно).

**Метрики.** MAE / RMSE / sMAPE / R². sMAPE предпочитается перед MAPE,
т.к. в данных встречаются нулевые наблюдения по редким парам
`(region, icd)`.

**Поток данных в продакшене.**

1. ETL запускается раз в квартал, когда МЗ РК публикует новые рецепты.
2. `prepare_data.py` агрегирует ~500 xlsx → один parquet `monthly_panel`.
3. `build_features.py` строит фичи; `train.py` переобучает LightGBM.
4. `metadata.json` фиксирует метрики и колонки — фронт сравнивает их с
   предыдущей версией.
5. Артефакт модели коммитится в `ml/models/`; FastAPI поднимается через
   `uvicorn backend.main:app`.
6. Фронт делает запросы `/api/forecast` → возвращает прогноз и
   рекомендацию.

## 3. Данные

### 3.1 Источник

- Платформа открытых данных `ashyq.data.gov.kz` (на базе Magda).
- 18 датасетов «Медицинская история граждан (<регион>)».
- 504 файла .xlsx (квартальные) + превью CSV (за 2024 Q3) на регион.
- Период: 2018 Q1 – 2024 Q3 (область Абай — до 2025 Q2).

### 3.2 Структура

| Колонка | Тип | Описание |
| --- | --- | --- |
| `recipedate` | datetime | Дата выдачи |
| `icdid` | str | Код МКБ-10 |
| `nozology` | str | Название диагноза |
| `region_med_organ` | str | Регион |
| `raion_med_organ` | str | Район |
| `recipepackqty` | int | Кол-во упаковок |
| `polyclid` | int | ID поликлиники |
| `dosage`, `medservicetype`, … | num | (служебные, не используются) |

### 3.3 Препроцессинг

- Очистка: drop NA по `recipedate / icdid / region_med_organ`.
- Strip whitespace, нормализация регистра.
- `recipepackqty` → numeric, NaN → 1 (рецепт без явного количества).
- Группировка: `(region, district, icdid, nozology, year_month)`
  → `recipe_count`, `total_packs`, `n_clinics`.
- Полная сетка месяцев на каждую серию (`region, icd`) — пропуски
  заполняются 0 (если нет рецепта — это значимая информация).
- Серии короче 18 месяцев исключаются из обучения (нечем рассчитать
  `lag_12` + рост).

### 3.4 Feature engineering

| Группа | Признаки |
| --- | --- |
| Лаги | `lag_1, lag_2, lag_3, lag_6, lag_12` |
| Скользящие | `roll_mean_{3,6,12}`, `roll_std_{3,6,12}` |
| Накопит. | `expanding_mean` |
| Регион | `region_total_lag1` (сумма по региону за прошлый месяц) |
| Календарь | `month`, `quarter`, `month_idx`, `month_sin/cos` |
| Категориаль. | `region_enc`, `icdid_enc`, `icd_chapter_enc` |
| Контекст | `n_clinics`, `n_districts` |

Целевая переменная — `log1p(recipe_count)`, на инференсе — обратное
преобразование `expm1`.

## 4. Модель

- **LightGBM** (`gbdt`), регрессия по `log1p(y)`.
- Параметры: `num_leaves=128`, `lr=0.05`, `min_data_in_leaf=40`,
  feature/bagging fraction = 0.9, `lambda_l2=1.0`, 900 раундов.
- Категориальные передаются через `categorical_feature` индексы.

### 4.1 Почему LightGBM

| Альтернатива | Минус для нашего кейса |
| --- | --- |
| Prophet / SARIMAX по каждой серии | 24 000+ серий, очень дорого; не учитывает кросс-серийные зависимости. |
| LSTM / Temporal Fusion Transformer | Длина историй 30–80 мес. — слишком короткая для сетей; высокая инфра-стоимость. |
| Линейная регрессия | Нелинейные сезонные взаимодействия с регионом и классом МКБ ловятся плохо. |
| **LightGBM** | Хорошо переносит разрежённые серии, нативно работает с категориалями, обучается за минуты. |

### 4.2 Бейзлайны

- `naive_last` — `y_{t-1}`.
- `seasonal_naive_12` — `y_{t-12}`.
- `rolling_mean_3` — среднее по 3 предыдущим месяцам.

### 4.3 Метрики (обновляются после `train.py`)

После запуска `python ml/train.py` метрики появляются в
`ml/models/metadata.json` и автоматически отображаются в карточке
«Качество модели» на фронте.

### 4.4 Многошаговый прогноз

`backend/inference.py::Forecaster.forecast()` рекурсивно: предсказание
для месяца _t+1_ добавляется к истории, затем пересчитываются лаги и
скользящие, и идёт прогноз _t+2_ — и так до `horizon` шагов.

## 5. Логика принятия решений (front-perspective)

Фронт строит **рекомендацию** на основе соотношения прогноза к среднему
за 12 предыдущих месяцев:

| Δ (прогноз vs среднее) | Тариф | Действие |
| --- | --- | --- |
| `< -15 %` | Резкий спад | Аудит причин (сбой регистрации?), перенаправить ресурсы. |
| `-15 % … -5 %` | Снижение | Сократить новые закупки, перераспределить запасы. |
| `±5 %` | Стабильно | Сохранить текущие объёмы. |
| `+5 % … +15 %` | Умеренный рост | Запас препаратов +10..20 %, проверить пропускную. |
| `> +15 %` | Резкий рост | Нарастить закупки, привлечь резерв медперсонала. |

Эта эвристика реализована в `frontend/components/ForecastPanel.tsx` —
её можно тонко подстроить под любые внутренние KPI МЗ РК.

## 6. Запуск

### 6.1 Данные → модель

```bash
# 1. Один раз — собрать данные (если ещё нет)
python -m pip install -r backend/requirements.txt python-calamine

# 2. ETL: ~30–60 мин на 12 ГБ данных
python ml/prepare_data.py

# 3. Feature engineering: <1 мин
python ml/build_features.py

# 4. Обучение: ~1–5 мин
python ml/train.py
```

### 6.2 Backend

```bash
cd backend
uvicorn main:app --host 0.0.0.0 --port 8000
# Документация Swagger: http://localhost:8000/docs
```

### 6.3 Frontend

```bash
cd frontend
npm install
npm run dev
# http://localhost:3000
```

`next.config.js` проксирует `/api/*` → `http://localhost:8000/api/*`.
Чтобы указать другой адрес backend:
```bash
NEXT_PUBLIC_API_URL=http://my-host:8000 npm run dev
```

## 8. Ограничения и улучшения

- Уровень района слабо подходит для классической регрессии — добавить
  иерархическую модель (региональная сеть с штрафом за разрежённые
  серии), либо пост-разверстать прогноз региона по долям районов.
- Внешние факторы (сезонные эпидемии гриппа, COVID-19, температура) не
  учитываются. Возможные источники: сводки ВОЗ, метеоданные.
- Можно использовать Bayesian quantile regression для интервалов
  доверия 80/95 %.
- В продакшене: A/B-сравнение по регионам перед раскаткой нового веса.
