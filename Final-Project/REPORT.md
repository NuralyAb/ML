> ML System Design · прогноз количества выписанных рецептов в Республике

## 1. Постановка задачи (Problem statement)

В Казахстане ежегодно выписываются миллионы рецептов на бесплатное и
льготное амбулаторное лекарственное обеспечение (АЛО). Министерство
здравоохранения РК сталкивается с задачами:

1. **Закупка** препаратов на следующий квартал по 18 регионам.
2. **Распределение кадров** — где ожидать рост обращений.
3. **Аудит поликлиник** — выявление аномального роста/падения выписки.

Все три задачи требуют прогноза _объёма выписки рецептов_ на 1–6 месяцев
вперёд **в разрезе диагноза и территории**. До сих пор такой прогноз
строился экстраполяцией прошлогодних значений вручную: это медленно и
не учитывает кросс-региональные шоки (COVID-19, сезонные эпидемии).

**Цель проекта:** автоматизированная ML-система прогнозирования
`recipe_count` — количества рецептов по `(region × icd_code × month)` —
с UI для МЗ РК и REST API для интеграции с дашбордами.

## 3. Новизна и оригинальность

- **Первая публичная** end-to-end система над датасетом
  ISLO_MEDICALHISTORYOFCITIZENS, охватывающая все 18 регионов РК
  (≈ 12 ГБ исходных xlsx) и обучаемая на _одной_ кросс-серийной модели.
- В отличие от типового подхода «один SARIMA на каждую серию», мы
  обучаем единую LightGBM-регрессию на всех сериях c географическими
  и нозологическими эмбеддингами через `LabelEncoder` + категориальный
  сплит. Это позволяет переносить сигнал между регионами и
  эффективно работать на разрежённых сериях редких диагнозов.
- Дашборд **переводит прогноз в управленческое решение**
  (5 уровней: drop / decline / stable / growth / surge) с
  человекочитаемой рекомендацией.
- **Фенотипирование серий через K-means.** Помимо supervised-прогноза,
  каждая `(region, icd)`-серия описывается 8 поведенческими признаками
  (уровень, CV, тренд, сезонность, доля нулей, ACF-1, ACF-12, длина) и
  кластеризуется в интерпретируемые группы (сезонные, растущие,
  волатильные, спорадические и т. п.). Полученный `cluster_id`
  возвращается в LightGBM как дополнительный категориальный признак —
  unsupervised сигнал помогает supervised модели.
- **Квантильные доверительные интервалы (P10–P90).** Дополнительные
  LightGBM-бустеры с `objective="quantile"` (alpha = 0.1 и 0.9) дают
  80%-й prediction interval на каждой точке прогноза. Дашборд рисует
  fan-chart (тёмная медиана + светлая лента); это превращает «прогноз»
  в **прогноз + диапазон риска**, что напрямую отвечает на ключевой
  вопрос закупок: «сколько брать страхового запаса?».
- **Audit-режим: детектор аномалий выписки.** Из трёх бизнес-задач
  §1 (закупки, кадры, **аудит клиник**) последняя оставалась незакрытой.
  Скрипт `ml/detect_anomalies.py` считает MAD-based z-score на
  log-residual hold-out-прогноза и ранжирует серии по `|z|`; дашборд
  показывает таблицу top-N и heatmap «регион × класс МКБ» с цветом
  по max|z|. Производственная модель теперь работает в трёх режимах:
  **прогноз / интервал / аудит**.
- **Self-service ingest через UI.** Эндпоинт `POST /api/ingest`
  принимает свежий xlsx (схема ЭРСБ) или parquet (схема monthly_panel),
  парсит его той же логикой, что bulk-ETL, дедуплицирует по
  `(region, district, icd, year_month)` и атомарно записывает
  `monthly_panel.parquet`. Дашборд имеет drop-зону: аналитик МЗ РК
  может закинуть очередной квартальный файл без CLI, и KPI/карта/heatmap
  обновятся за секунды. Прогноз и аномалии — на офлайн-цикле
  (правильное разделение MLops: data ingest = частая операция,
  retraining = редкая).

## 5. Методология

### 5.1 Источник и сбор данных

- 18 датасетов «Медицинская история граждан» по регионам РК.
- 504 квартальных xlsx-файла, период **2018 Q1 – 2024 Q3**
  (Абай: до 2025 Q2).
- Скачивание автоматизировано через Magda API (см.
  `datasets/_manifest.tsv` и описание в `README.md`).

### 5.2 Препроцессинг (`ml/prepare_data.py`)

1. Чтение xlsx через `python-calamine` (быстрее и легче по памяти, чем
   openpyxl) с фильтром колонок:
   `recipedate, icdid, nozology, region_med_organ, raion_med_organ,
    recipepackqty, polyclid`.
2. Drop NA по ключевым полям; нормализация регистра/whitespace.
3. Группировка `(region, district, icdid, nozology, year_month)` →
   `recipe_count`, `total_packs`, `n_clinics`.
4. Сохранение в parquet — итоговый размер ≈ 50 МБ
   (vs. 12 ГБ исходных).

### 5.3 Feature engineering (`ml/build_features.py`)

- **Уровень модели** — `(region × icd × month)`.
- **Полная сетка месяцев** — пропуски заполняются 0.
- **Лаги:** 1, 2, 3, 6, 12.
- **Скользящие:** mean/std по 3, 6, 12 месяцам.
- **Накопит. среднее:** `expanding_mean`.
- **Регион:** `region_total_lag1`.
- **Календарь:** `month, quarter, month_idx, sin/cos(month)`.
- **ICD chapter:** буква (A..Z) — широкий класс заболеваний.
- **Категориальные:** `region, icdid, icd_chapter` через label-encode +
  `categorical_feature` LightGBM.
- **Серии < 18 месяцев** исключаются.

### 5.4 Модель (`ml/train.py`)

| Гиперпараметр | Значение |
| --- | --- |
| objective | regression |
| target transform | `log1p(y)` (count distribution с длинным хвостом) |
| learning_rate | 0.05 |
| num_leaves | 128 |
| min_data_in_leaf | 40 |
| feature_fraction | 0.9 |
| bagging_fraction / freq | 0.9 / 5 |
| lambda_l2 | 1.0 |
| num_boost_round | 900 |

### 5.5 Метрики

- **MAE** — устойчивая к выбросам.
- **RMSE** — штрафует крупные промахи (важно для бюджетного риска).
- **sMAPE** — относительная ошибка, симметричная и устойчивая к нулям.
- **R²** — доля объяснённой дисперсии.

### 5.6 Бейзлайны

| Модель | Описание |
| --- | --- |
| `naive_last` | `y_{t-1}` |
| `seasonal_naive_12` | `y_{t-12}` |
| `rolling_mean_3` | среднее по 3 предыдущим |
| **LightGBM** | модель выше |

### 5.7 Валидация

Time-based hold-out: последние **6 месяцев** на каждую серию — тест.
Итоговая модель обучается на всех данных кроме этих 6 месяцев. Это
эквивалентно walk-forward-валидации без пересборки фичей.

### 5.8 Quantile-прогнозирование (`ml/train_quantile.py`)

Точечный прогноз отвечает на «сколько ожидать», но не отвечает на
**«сколько закупить страхового запаса, чтобы дефицит был редким?»**.
LightGBM из коробки поддерживает `objective="quantile"`, поэтому мы
обучаем **две дополнительные модели** на тех же 23 признаках и том же
`log1p(y)`-таргете:

- `lgbm_recipe_forecast_q10.txt` — `alpha = 0.1` → пессимистическая
  оценка (P10).
- `lgbm_recipe_forecast_q90.txt` — `alpha = 0.9` → оптимистическая
  оценка (P90).

Так как квантили инвариантны к монотонным преобразованиям,
квантили в исходных единицах получаются через `expm1`. На каждой
точке рекурсивного forecast'а API возвращает тройку
`{point, lower, upper}`, фронт рисует **fan-chart** (тёмная средняя
линия + полупрозрачная амбер-лента 80%-го интервала).

**Калибровка на hold-out**: empirical coverage **78.57 %** (цель 80 %,
отклонение −1.4 п.п. — модель калибрована хорошо), pinball-loss
(log-scale): q10 = 0.0273, q90 = 0.0471, mean interval width = 44.7
рецепта, median = 2.02 (длинный хвост сжимает median).
MLflow-эксперимент `10_lgbm_quantile` хранит все метрики и сами
бустеры. На рекурсию подаётся **точечный прогноз** (не P10/P90),
чтобы лента не смещалась оптимистично или пессимистично.

### 5.9 Аудит выписки (`ml/detect_anomalies.py`)

Третья задача из §1 — **поиск аномальной выписки в прошлых
периодах** — закрыта отдельным модулем. На hold-out наборе
(последние 6 месяцев каждой серии) считаем остатки в log-пространстве:

```
r = log1p(actual) − log1p(predicted)
σ̂ = 1.4826 · MAD(r)                # robust-оценка σ для гауссова шума
z = (r − median(r)) / σ̂
```

log-преобразование снимает доминирование высокообъёмных серий, MAD
устойчив к выбросам, а коэффициент 1.4826 делает оценку
сопоставимой со стандартным отклонением нормального распределения.
Severity buckets:

- `|z| ≥ 3.0` → **critical** (требует срочного аудита),
- `2.0 ≤ |z| < 3.0` → **warning**,
- `1.5 ≤ |z| < 2.0` → **notice**,
- иначе → нормально (не сохраняется).

**Направление** определяется знаком `z`: `surge` (выписка выше
ожидаемой — потенциальный эпид-всплеск или мошенничество) vs
`drop` (выписка ниже ожидаемой — потенциальный сбой регистрации
или реальное снижение заболеваемости). Артефакт
`ml/data/anomalies.parquet` отдаётся через эндпоинты
`/api/anomalies` (top-N + фильтры) и `/api/anomaly-heatmap`
(max|z| по ячейкам `регион × класс МКБ`). Дашборд использует обе
визуализации — таблица + цветовая карта.

### 5.10 Unsupervised фенотипирование серий (`ml/cluster_series.py`)

Supervised-стек учит одну глобальную регрессию на гетерогенной популяции
серий. Чтобы явно описать структуру этой популяции, мы запускаем
**K-means на per-series поведенческих признаках** — это шаг
ML-разведки и одновременно источник дополнительного признака для
прогноза.

**Признаки серии** (8 чисел на каждую `(region, icd)`):

| Признак | Что описывает |
| --- | --- |
| `level` | `log1p(mean)` — типичный объём выписки |
| `cv` | std/mean — относительная волатильность |
| `trend` | OLS-наклон ÷ среднее × длина — относительный рост за всю историю |
| `seasonality` | размах среднемесячных значений / общее среднее |
| `share_zeros` | доля месяцев с 0 рецептов |
| `acf_1` | автокорреляция лаг-1 — гладкость |
| `acf_12` | автокорреляция лаг-12 — годовая сезонная устойчивость |
| `n_months` | длина серии |

**Пайплайн.** `StandardScaler` → перебор `k ∈ {3..8}` →
выбор по максимуму `silhouette score` (более надёжный критерий, чем
elbow на инерции при умеренной размерности) → KMeans (`n_init=10`,
`random_state=42`).

**Именование кластеров.** Каждый кластер получает читаемое имя по
доминирующей координате стандартизованного центроида (наибольший
|z-score|) — например `seasonal`, `growing`, `sporadic`,
`high_volume`, `volatile`. Это нужно только для отчётности; источник
истины — числовой `cluster_id`.

**Выход:** `ml/data/series_clusters.parquet`,
`ml/data/cluster_profiles.csv`, `ml/models/cluster_pca.png`
(PCA-2D scatter), MLflow-эксперимент `series_clustering`
с метриками `silhouette`, `inertia`, `calinski_harabasz`.

**Интеграция в прогноз.** Отдельный MLflow-эксперимент
`9_lgbm_with_cluster` повторяет baseline LightGBM, но добавляет
`cluster_id` в категориальные признаки. Сравнение vs `1_lgbm_baseline`
в одной MLflow-таблице показывает, несёт ли unsupervised-фенотип
сигнал, которого не было в lag/rolling/calendar.

                │     (MAD-z-scores on holdout → /api/anomalies, /api/anomaly-heatmap)

### 6.1 Логика на проде

1. **Online API.** МЗ РК делает HTTP-запрос
   `POST /api/forecast {region, icd, horizon}`. Сервис возвращает
   историю + прогноз + рекомендацию.
2. **Дашборд.** Аналитик выбирает регион/диагноз; UI показывает
   карту, тренд и прогноз.
3. **Офлайн-обучение.** ETL+train запускается раз в квартал (после
   публикации новых рецептов в открытом портале) или через
   pull-request в CI.
4. **Мониторинг.** `metadata.json` содержит метрики; раз в месяц
   сравниваются текущая и предыдущая версии — при деградации MAE
   срабатывает алерт.

### 6.2 Принятие решений

| Δ (прогноз vs средние 12 мес.) | Уровень | Решение |
| --- | --- | --- |
| `> +15 %` | surge | срочно нарастить запасы, привлечь резерв медперсонала |
| `+5 %..+15 %` | growth | страховой запас +10..20 %, проверка пропускной способности |
| `±5 %` | stable | сохранить текущие объёмы |
| `-15 %..-5 %` | decline | сократить новые закупки, перераспределить запасы |
| `< -15 %` | drop | аудит причин (сбой регистрации? снижение заболеваемости?) |

### Аудит выписки (§5.9)

Робастные параметры MAD-z-score на hold-out:
`median(residual_log) = 0.0`, `MAD = 0.0974`, `σ̂ = 0.1444`.
На **113 286 holdout-строк** скрипт пометил **34 843** (30.8 %):

### Unsupervised-фенотип серий

K-means на 8 поведенческих признаках (см. §5.8) разделяет
**18 881 серию** `(region, icd)` на **k = 3** интерпретируемые
группы. Выбор `k` сделан по силуэту — он монотонно убывает на сетке
`k ∈ {3..8}` (см. ниже), что указывает на естественную трёхуровневую
структуру данных.

| k | silhouette | inertia | calinski-harabasz |
| ---: | ---: | ---: | ---: |

| 4 | 0.187 | 72 814 | 6 761 |
| 5 | 0.190 | 66 407 | 6 015 |

Профили кластеров (центроиды в исходных единицах):

| id | имя | размер | доля | level | cv | trend | seasonality | share_zeros | acf_1 | acf_12 | n_months |

| 2 | non_seasonal_flat | 8 139 | 43.1 % | 1.19 | 1.32 | 0.22 | 1.96 | 0.42 | 0.20 | 0.05 | 60.0 |

Содержательная трактовка (top-5 примеров серий в каждом кластере
см. в `ml/data/cluster_profiles.csv`):

- **`high_volume`** — хронические массовые диагнозы:
  артериальная гипертензия (I11.9), ишемическая болезнь сердца
  (I20.8), сахарный диабет (E11.7 – E11.9). Длинные ряды (~73 мес.),
  низкая волатильность, выраженная автокорреляция и лаг-1, и лаг-12.
- **`seasonal`** — на самом деле «спорадически-сезонные»: высокая
  относительная сезонность (4.27) **при доле нулей 0.76** —
  редкие нозологии с разрозненными всплесками активности
  (онкология C67.0, F53.0 послеродовые психозы, N08.0
  гломерулярные заболевания).
- **`non_seasonal_flat`** — «середняки»: умеренный объём, умеренная
  волатильность, нет ни сильной сезонности, ни роста.

PCA-проекция (PC1 = 52.5 %, PC2 = 12.1 %, итого 64.6 % дисперсии в
2D) — `ml/models/cluster_pca.png` — визуально подтверждает три чётких
региона в feature-space.

**Эксперимент `9_lgbm_with_cluster` vs `1_lgbm_baseline`** (один и
тот же hold-out, единственное отличие — наличие `cluster_id` в
категориальных признаках):

| Метрика | baseline | + cluster_id | Δ |
| --- | ---: | ---: | ---: |
| MAE | **18.68** | 18.92 | +0.24 (хуже) |
| RMSE | 278.79 | **276.17** | −2.62 (лучше) |
| sMAPE | 64.3 % | **64.0 %** | −0.3 pp (лучше) |
| R² | 0.9377 | **0.9388** | +0.0011 (лучше) |
| time | 18.3 s | 22.6 s | +4.3 s |

Различия — в пределах шума. `cluster_id` даёт мизерный выигрыш на
хвосте ошибок (RMSE, R²) и чуть ухудшает медианную точность (MAE).
Это **диагностически чистый результат**: lag/rolling/categorical
фичи через сплиты по `(region, icdid, icd_chapter)` уже неявно
кодируют большую часть фенотипа, и явный 3-уровневый `cluster_id`
несёт лишь ~2 бита информации поверх. Кластеризация остаётся
ценной **как описательный артефакт** (профили + PCA-плот для отчёта
и дашборда), но не как фича для прогноза.

### Ограничения

- Уровень _района_ слишком разрежён для классической регрессии.
- Внешние факторы (гриппозные волны, температура, эпидемии) не
  моделируются.
- ~~Нет интервалов доверия~~ → закрыто в §5.8 (P10/P90 quantile-бустеры,
  fan-chart на дашборде).

### Возможные улучшения

1. Иерархическое согласование (`hts` / Mint reconciliation).
2. Добавить погоду и сводки ВОЗ как экзогенные ряды.
3. Bayesian quantile regression → 80/95 % интервалы для бюджета.
4. CI: ежеквартальный авто-train + smoke-tests на front-перцентили.

## 8. Визуализация

Дашборд (Next.js) включает:

1. **KPI-плитки**: регионов, районов, МКБ-кодов, всего рецептов.
2. **Карта Казахстана** — схематичная сетка 5×4, цвет = log(рецептов),
   клик = выбор региона.
3. **Бары регионов** за 12 мес., сортировка по убыванию.
4. **Heatmap регион × класс МКБ** — найти регионы с аномальной
   нагрузкой по конкретному классу болезней.
5. **Топ-15 диагнозов** в выбранном регионе (или по стране).
6. **Forecast-панель** — выбор региона, МКБ-кода, горизонта (1/3/6/12);
   график «факт + прогноз», таблица помесячного прогноза, плитка с
   рекомендацией системы.
7. **Карточка модели** — таблица метрик, топ-признаков, scatter
   «факт vs прогноз» на hold-out.
