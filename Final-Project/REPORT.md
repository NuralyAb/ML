# Med Forecast KZ — итоговый отчёт

> ML System Design · прогноз количества выписанных рецептов в Республике
> Казахстан в разрезе регион × район × диагноз (МКБ-10)
> на основе открытых данных ЭРСБ МЗ РК.

**Демо UI:** Next.js — `/`, FastAPI Swagger — `/docs`.

---

## 1. Постановка задачи (Problem statement)

В Казахстане ежегодно выписываются миллионы рецептов на бесплатное и
льготное амбулаторное лекарственное обеспечение (АЛО). Министерство
здравоохранения РК сталкивается с задачами:

1. **Закупка** препаратов на следующий квартал по 18 регионам.
2. **Распределение кадров** — где ожидать рост обращений.
3. **Аудит поликлиник** — выявление аномального роста/падения выписки.

Все три задачи требуют прогноза _объёма выписки рецептов_ на 1–6 месяцев
вперёд **в разрезе диагноза и территории**. До сих пор такой прогноз
строился экстраполяцией прошлогодних значений вручную: это медленно и
не учитывает кросс-региональные шоки (COVID-19, сезонные эпидемии).

**Цель проекта:** автоматизированная ML-система прогнозирования
`recipe_count` — количества рецептов по `(region × icd_code × month)` —
с UI для МЗ РК и REST API для интеграции с дашбордами.

| Медицинский | Ранняя сигнализация эпидемий и аномалий выписки. |

## 3. Новизна и оригинальность

- **Первая публичная** end-to-end система над датасетом
  ISLO_MEDICALHISTORYOFCITIZENS, охватывающая все 18 регионов РК
  (≈ 12 ГБ исходных xlsx) и обучаемая на _одной_ кросс-серийной модели.
- В отличие от типового подхода «один SARIMA на каждую серию», мы
  обучаем единую LightGBM-регрессию на всех сериях c географическими
  и нозологическими эмбеддингами через `LabelEncoder` + категориальный
  сплит. Это позволяет переносить сигнал между регионами и
  эффективно работать на разрежённых сериях редких диагнозов.
- Дашборд **переводит прогноз в управленческое решение**
  (5 уровней: drop / decline / stable / growth / surge) с
  человекочитаемой рекомендацией.

## 5. Методология

### 5.1 Источник и сбор данных

- 18 датасетов «Медицинская история граждан» по регионам РК.
- 504 квартальных xlsx-файла, период **2018 Q1 – 2024 Q3**
  (Абай: до 2025 Q2).
- Скачивание автоматизировано через Magda API (см.
  `datasets/_manifest.tsv` и описание в `README.md`).

### 5.2 Препроцессинг (`ml/prepare_data.py`)

1. Чтение xlsx через `python-calamine` (быстрее и легче по памяти, чем
   openpyxl) с фильтром колонок:
   `recipedate, icdid, nozology, region_med_organ, raion_med_organ,
    recipepackqty, polyclid`.
2. Drop NA по ключевым полям; нормализация регистра/whitespace.
3. Группировка `(region, district, icdid, nozology, year_month)` →
   `recipe_count`, `total_packs`, `n_clinics`.
4. Сохранение в parquet — итоговый размер ≈ 50 МБ
   (vs. 12 ГБ исходных).

### 5.3 Feature engineering (`ml/build_features.py`)

- **Уровень модели** — `(region × icd × month)`.
- **Полная сетка месяцев** — пропуски заполняются 0.
- **Лаги:** 1, 2, 3, 6, 12.
- **Скользящие:** mean/std по 3, 6, 12 месяцам.
- **Накопит. среднее:** `expanding_mean`.
- **Регион:** `region_total_lag1`.
- **Календарь:** `month, quarter, month_idx, sin/cos(month)`.
- **ICD chapter:** буква (A..Z) — широкий класс заболеваний.
- **Категориальные:** `region, icdid, icd_chapter` через label-encode +
  `categorical_feature` LightGBM.
- **Серии < 18 месяцев** исключаются.

### 5.4 Модель (`ml/train.py`)

| Гиперпараметр | Значение |
| --- | --- |
| objective | regression |
| target transform | `log1p(y)` (count distribution с длинным хвостом) |
| learning_rate | 0.05 |
| num_leaves | 128 |
| min_data_in_leaf | 40 |
| feature_fraction | 0.9 |
| bagging_fraction / freq | 0.9 / 5 |
| lambda_l2 | 1.0 |
| num_boost_round | 900 |

### 5.5 Метрики

- **MAE** — устойчивая к выбросам.
- **RMSE** — штрафует крупные промахи (важно для бюджетного риска).
- **sMAPE** — относительная ошибка, симметричная и устойчивая к нулям.
- **R²** — доля объяснённой дисперсии.

### 5.6 Бейзлайны

| Модель | Описание |
| --- | --- |
| `naive_last` | `y_{t-1}` |
| `seasonal_naive_12` | `y_{t-12}` |
| `rolling_mean_3` | среднее по 3 предыдущим |
| **LightGBM** | модель выше |

### 5.7 Валидация

Time-based hold-out: последние **6 месяцев** на каждую серию — тест.
Итоговая модель обучается на всех данных кроме этих 6 месяцев. Это
эквивалентно walk-forward-валидации без пересборки фичей.

## 6. Системная архитектура

```
       ashyq.data.gov.kz                           NEXT.JS DASHBOARD
                │                                ┌───────────────────┐
   504 xlsx (12GB)                               │  KPI · карта · heatmap │
                │                                │  Forecast UI · scatter │
                ▼                                └─────────▲─────────┘
        ml/prepare_data.py                                 │ /api/*
        (calamine, multiprocess)                           │
                │                            ┌─────────────┴─────────────┐
                ▼                            │      FastAPI · DuckDB     │
        monthly_panel.parquet (~50MB) ─────► │  читает parquet zero-copy │
                │                            └─────────────▲─────────────┘
                ▼                                          │
        ml/build_features.py ──► feature_panel.parquet ────┘
                │
                ▼
        ml/train.py → lgbm_recipe_forecast.txt + metadata.json
                │
                ▼
        backend/inference.py (recursive multi-step forecast)
```

### 6.1 Логика на проде

1. **Online API.** МЗ РК делает HTTP-запрос
   `POST /api/forecast {region, icd, horizon}`. Сервис возвращает
   историю + прогноз + рекомендацию.
2. **Дашборд.** Аналитик выбирает регион/диагноз; UI показывает
   карту, тренд и прогноз.
3. **Офлайн-обучение.** ETL+train запускается раз в квартал (после
   публикации новых рецептов в открытом портале) или через
   pull-request в CI.
4. **Мониторинг.** `metadata.json` содержит метрики; раз в месяц
   сравниваются текущая и предыдущая версии — при деградации MAE
   срабатывает алерт.

### 6.2 Принятие решений

| Δ (прогноз vs средние 12 мес.) | Уровень | Решение |
| --- | --- | --- |
| `> +15 %` | surge | срочно нарастить запасы, привлечь резерв медперсонала |
| `+5 %..+15 %` | growth | страховой запас +10..20 %, проверка пропускной способности |
| `±5 %` | stable | сохранить текущие объёмы |
| `-15 %..-5 %` | decline | сократить новые закупки, перераспределить запасы |
| `< -15 %` | drop | аудит причин (сбой регистрации? снижение заболеваемости?) |

| Rolling mean (3 мес.) | … | … | … | … |

Топ-признаков по gain (выводится в дашборде): `lag_1, roll_mean_3,
roll_mean_12, lag_12, region_total_lag1`. Это согласуется с
интуицией: текущий уровень определяется недавней историей, сезонной
базой и общим объёмом выписки в регионе.

### Интерпретация

- LightGBM устойчиво обыгрывает все три бейзлайна на sMAPE и R².
- `lag_12` важен для сезонных классов (J — респираторные заболевания
  с зимним пиком).
- `region_total_lag1` помогает «делить» общий региональный шок между
  диагнозами (например, COVID-волны 2020–2021).

### Ограничения

- Уровень _района_ слишком разрежён для классической регрессии.
- Внешние факторы (гриппозные волны, температура, эпидемии) не
  моделируются.
- Нет интервалов доверия — будет добавлено через quantile LightGBM.

### Возможные улучшения

1. Иерархическое согласование (`hts` / Mint reconciliation).
2. Добавить погоду и сводки ВОЗ как экзогенные ряды.
3. Bayesian quantile regression → 80/95 % интервалы для бюджета.
4. CI: ежеквартальный авто-train + smoke-tests на front-перцентили.

## 8. Визуализация

Дашборд (Next.js) включает:

1. **KPI-плитки**: регионов, районов, МКБ-кодов, всего рецептов.
2. **Карта Казахстана** — схематичная сетка 5×4, цвет = log(рецептов),
   клик = выбор региона.
3. **Бары регионов** за 12 мес., сортировка по убыванию.
4. **Heatmap регион × класс МКБ** — найти регионы с аномальной
   нагрузкой по конкретному классу болезней.
5. **Топ-15 диагнозов** в выбранном регионе (или по стране).
6. **Forecast-панель** — выбор региона, МКБ-кода, горизонта (1/3/6/12);
   график «факт + прогноз», таблица помесячного прогноза, плитка с
   рекомендацией системы.
7. **Карточка модели** — таблица метрик, топ-признаков, scatter
   «факт vs прогноз» на hold-out.
